# OfferBot 产品需求文档（PRD）

> 版本: 1.0 · 2026.04
> 状态: V1 已实现

## 一、产品概述

| 项目 | 内容 |
|------|------|
| 产品名称 | OfferBot |
| 产品类型 | AI 求职顾问 Agent（本地部署，开源） |
| 目标用户 | 互联网行业求职者（技术/产品/运营） |
| 核心价值 | 通过 AI Agent 覆盖求职全流程，减少重复劳动，提升匹配效率 |
| 交互方式 | 对话驱动 + 场景引导卡片 + 补充视图面板 |
| 部署方式 | 本地运行，数据不出设备 |

## 三、核心功能详述

### F1: 多平台数据采集

| 项目 | 内容 |
|------|------|
| 优先级 | P0 |
| 用户故事 | 作为求职者，我希望 Agent 能从猎聘和智联自动搜索和抓取岗位信息，这样我可以获得跨平台的岗位数据 |
| 功能描述 | 通过 getjob 服务（Java）执行浏览器自动化，AI Agent 通过 HTTP API 控制爬取任务 |

**功能要点：**
- 支持猎聘和智联两个平台
- AI 根据用户画像自动生成搜索配置（关键词、城市、薪资范围）
- 支持"仅爬取"模式（scrapeOnly），先收集数据再决定投递
- 爬取完成后自动同步到本地数据库
- 爬取完成后自动构建知识图谱（LightRAG）

**用户流程：**
```
用户: "帮我看看猎聘上有什么 AI 岗位"
  → Agent 检查 getjob 服务状态和登录状态
  → Agent 根据用户画像生成搜索配置
  → 展示配置摘要，等待用户确认
  → 启动爬取任务（scrapeOnly 模式）
  → 后台轮询进度，完成后通知用户
  → 自动同步数据到本地 + 构建知识图谱
  → "已爬取 156 个岗位，要我帮你分析匹配度吗？"
```

### F2: 知识图谱检索

| 项目 | 内容 |
|------|------|
| 优先级 | P0 |
| 用户故事 | 作为求职者，我希望能用自然语言搜索岗位，不仅是关键词匹配，还能理解语义 |
| 功能描述 | 基于 LightRAG 的知识图谱 + 向量混合检索，支持语义推荐和聚合分析 |

**两种检索模式：**

| 模式 | 触发场景 | 示例 | 返回 |
|------|----------|------|------|
| answer | 需要分析/推荐/问答 | "适合我的岗位"、"AI 岗位技能 TOP10" | 文本回答（Agent 消费） |
| search | 需要岗位列表 | "找跟这个类似的岗位" | 岗位精简列表（展示给用户） |

**与 SQL 查询的路由规则：**
- 用户提到具体筛选条件（城市、薪资、公司名）→ `query_jobs`（SQL）
- 用户问"适合我的"、"推荐"、"类似的"、"差距"→ `rag_query`（知识图谱）

### F3: 7 维度匹配分析

| 项目 | 内容 |
|------|------|
| 优先级 | P0 |
| 用户故事 | 作为求职者，我希望 Agent 能量化分析我和每个岗位的匹配程度，告诉我差距在哪 |
| 功能描述 | 结合程序化计算和 LLM 评估，从 7 个维度给出匹配分数 |

**匹配维度：**

| 维度 | 权重 | 计算方式 | 说明 |
|------|------|----------|------|
| 技能匹配 | 30% | LLM 评估 | 用户技能 ∩ 岗位要求技能 |
| 经验匹配 | 15% | 程序化 | 用户年限 vs 岗位要求区间 |
| 薪资匹配 | 15% | 程序化 | 用户期望 vs 岗位薪资重叠度 |
| 位置匹配 | 10% | 程序化 | 城市是否在目标列表中 |
| 学历匹配 | 10% | 程序化 | 用户学历 ≥ 岗位要求 |
| 职责匹配 | 10% | LLM 评估 | 用户经验 vs 岗位职责 |
| 语义相似度 | 10% | Embedding | 简历与 JD 的向量相似度 |

**输出：**
- 综合匹配分数（0-100）
- 各维度分数和分析说明
- 缺失技能列表（技能差距分析）
- 匹配技能列表

### F4-F6: 投递与沟通

| 功能 | 优先级 | 描述 |
|------|--------|------|
| F4: 岗位统计 | P0 | 投递数量、回复率、按城市/行业/薪资分布 |
| F5: 打招呼语 | P1 | 基于用户画像 + JD 生成个性化内容，≤200 字 |
| F6: 智能投递 | P1 | 匹配度达标 → 生成打招呼语 → 用户确认 → 执行投递 |

**投递流程的关键设计决策：**
- 投递前必须经过用户确认（不做无人值守的自动投递）
- 支持"仅爬取"模式，先分析再投递
- 每次投递间隔 10-30 秒随机延迟（反检测）

### F7: 面试漏斗追踪

| 项目 | 内容 |
|------|------|
| 优先级 | P0 |
| 用户故事 | 作为求职者，我同时投递多个岗位，希望有一个全局视角管理所有面试进度 |
| 功能描述 | 10 阶段状态机 + 漏斗转化率分析 |

**状态机：**
```
applied → viewed → replied → interview_scheduled
  → round_1 → round_2 → round_3 → hr_round → offer
  （任意阶段可转入 rejected 或 withdrawn）
```

**漏斗指标：**
- 投递→已读率
- 已读→回复率
- 回复→约面率
- 约面→offer 率

### F8: 记忆画像系统

| 项目 | 内容 |
|------|------|
| 优先级 | P1（但对长期价值至关重要） |
| 用户故事 | 作为求职者，我希望 OfferBot 能记住我在对话中提到的各种信息，越用越了解我 |
| 功能描述 | 10 维度记忆分类，对话后异步提取，持久化到本地文件 |

**10 个记忆维度：**

| 维度 | 说明 | 应用场景 |
|------|------|----------|
| personal_thoughts | 个人想法 | 面试准备 |
| job_sprint_goals | 求职冲刺目标 | 搜索配置生成 |
| language_style | 语言风格 | 打招呼语生成 |
| personality_traits | 性格特征 | 面试准备 |
| hobbies_interests | 兴趣爱好 | 打招呼语 |
| career_planning | 职业规划 | 匹配分析 |
| personal_needs | 个人需求 | 岗位筛选 |
| key_points | 要点信息 | 面试准备 |
| communication_preferences | 沟通偏好 | 打招呼语 |
| values_beliefs | 价值观 | 匹配分析 |

**记忆生命周期：**
```
对话中用户透露信息 → 对话结束触发 → MemoryExtractor（独立 LLM 调用）
→ 分析对话 + 对比已有记忆 → save/update/delete → 写入本地 Markdown 文件
→ 下次对话启动时加载到 System Prompt
```

### F9-F10: 简历与面试

| 功能 | 优先级 | 描述 |
|------|--------|------|
| F9: 简历解析 | P1 | 支持 PDF/Markdown 上传，提取结构化字段，生成 DOCX |
| F10: 面试准备 | P1 | 基于用户记忆 + 岗位 JD 生成针对性面试建议 |

### F11-F13: 系统能力

| 功能 | 优先级 | 描述 |
|------|--------|------|
| F11: 多 LLM 适配 | P0 | 支持 DashScope/OpenAI/Gemini/DeepSeek，配置切换 |
| F12: Skills 系统 | P1 | 场景参考规范，文件夹架构，按需加载 |
| F13: Web 工具 | P2 | 网页抓取 + 搜索，获取公司信息/技术文档等 |

## 四、交互设计

### 4.1 核心交互模式

OfferBot 采用**对话驱动 + 场景引导**的交互模式：

```
┌─────────────────────────────────────────────────┐
│  OfferBot                                        │
│                                                  │
│  ┌──────────────────────────────────────────┐   │
│  │  欢迎回来！你有 3 个岗位等待面试状态更新。  │   │
│  │                                          │   │
│  │  你可以试试：                             │   │
│  │  ┌──────────┐ ┌──────────┐ ┌──────────┐ │   │
│  │  │ 推荐岗位  │ │ 查看进度  │ │ 面试准备  │ │   │
│  │  └──────────┘ └──────────┘ └──────────┘ │   │
│  └──────────────────────────────────────────┘   │
│                                                  │
│  [对话区域 - 多轮对话 + 工具调用过程展示]         │
│                                                  │
│  ┌──────────────────────────────────────────┐   │
│  │  输入消息...                    [发送]    │   │
│  └──────────────────────────────────────────┘   │
│                                                  │
│  [侧边栏: 岗位列表 | 统计面板 | 面试追踪]        │
└─────────────────────────────────────────────────┘
```

### 4.2 场景引导卡片

首次使用时展示的引导卡片：

| 卡片 | 对应指令 | 覆盖场景 |
|------|----------|----------|
| "帮我搜索岗位" | 触发数据采集流程 | S2 |
| "推荐适合我的岗位" | 触发知识图谱推荐 | S3 |
| "分析这个岗位的匹配度" | 触发匹配分析 | S4 |
| "查看投递进度" | 触发统计查询 | S6 |
| "帮我准备面试" | 触发面试准备 | 面试准备 Skill |

### 4.3 补充视图面板

对话界面之外的数据展示面板：

| 面板 | 内容 | 触发方式 |
|------|------|----------|
| 岗位列表 | 表格展示，支持排序/筛选 | 对话中提到岗位列表时自动切换 |
| 统计面板 | 投递数量、回复率、面试漏斗 | 对话中查看统计时自动切换 |
| 简历管理 | 上传/查看/下载简历 | 手动切换 |
| 设置页面 | API Key、匹配阈值、目标配置 | 手动切换 |

## 五、数据模型

### 5.1 核心实体关系

```
用户 (resumes + job_preferences + 记忆画像)
  │
  ├── 1:N → 岗位 (jobs)
  │           │
  │           ├── 1:1 → 匹配分析 (match_results)
  │           │
  │           └── 1:1 → 投递记录 (applications)
  │                       │
  │                       └── 1:N → 面试追踪 (interview_tracking + stage_log)
  │
  ├── 1:N → 对话历史 (conversations + JSONL)
  │
  └── 1:N → 记忆条目 (记忆画像/*.md)
```

### 5.2 关键数据流

```
外部平台 → getjob 爬取 → 同步到本地 jobs 表 → 构建知识图谱
                                    ↓
用户画像 + 岗位数据 → AI 匹配分析 → match_results 表
                                    ↓
匹配通过的岗位 → 生成打招呼语 → 用户确认 → 执行投递 → applications 表
                                                          ↓
                                              面试状态更新 → interview_tracking 表
                                                          ↓
                                              漏斗分析 → 策略调整建议
```

## 六、非功能需求

| 维度 | 要求 |
|------|------|
| 性能 | 单次对话响应 < 30 秒（含 LLM 调用）；批量匹配分析 10 个岗位 < 3 分钟 |
| 安全 | 所有数据本地存储；API Key 不明文展示；无数据上传行为 |
| 可用性 | 对话式交互，零学习成本；场景引导卡片降低使用门槛 |
| 可扩展性 | Skills 插件体系支持社区贡献新场景；多 LLM 供应商可配置切换 |
| 可靠性 | 工具调用失败自动重试（3 次）；爬取任务中断可恢复 |
| 兼容性 | 支持 macOS / Linux；Python 3.11+ |
