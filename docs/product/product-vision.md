# OfferBot 产品愿景与定位

> 版本: 1.0 · 2026.04

## 一、产品背景

### 市场痛点

中国互联网求职市场存在一个结构性矛盾：**求职者的时间和精力是有限的，但求职流程的复杂度在持续增加。**

一个典型的求职者在一次求职周期中需要完成以下动作：

| 环节 | 典型耗时 | 痛点 |
|------|----------|------|
| 浏览岗位 | 每天 1-2 小时 | 信息过载，大量不匹配的岗位混杂其中 |
| 分析 JD | 每个岗位 5-10 分钟 | 需要人工判断匹配度，容易遗漏关键信息 |
| 定制打招呼语 | 每个岗位 3-5 分钟 | 千篇一律的模板导致回复率低 |
| 投递操作 | 每个岗位 1-2 分钟 | 重复性机械操作，跨平台切换繁琐 |
| 面试追踪 | 每天 15-30 分钟 | 多岗位并行时容易混乱，缺乏全局视角 |
| 面试准备 | 每个岗位 1-3 小时 | 缺乏针对性，不知道该重点准备什么 |

一个活跃求职者每天在求职上花费 3-5 小时，其中超过 70% 是重复性、低创造性的工作。

### 现有方案的不足

| 方案 | 能力 | 局限 |
|------|------|------|
| Boss 直聘/猎聘/智联 | 岗位搜索、简单推荐 | 推荐算法黑盒，无法基于用户真实经验做深度匹配 |
| 脉脉 | 社交求职、公司评价 | 信息碎片化，无法形成系统的求职策略 |
| 简历优化工具 | 简历模板、AI 润色 | 只解决简历环节，不覆盖全流程 |
| 求职 Excel 表格 | 手动记录投递状态 | 纯人工维护，无智能分析能力 |
| 自动投递脚本 | 批量投递 | 无差别投递，无匹配分析，容易被平台封禁 |

核心问题：**没有一个产品能以"理解用户"为基础，覆盖从岗位发现到拿 offer 的完整链路。**

### 市场机会

2025-2026 年，大语言模型（LLM）的能力已经成熟到可以：
- 理解非结构化的 JD 文本并提取结构化信息
- 基于用户画像做多维度匹配分析
- 生成个性化的沟通内容
- 通过对话交互完成复杂的多步骤任务

这意味着一个 AI Agent 可以扮演"私人求职顾问"的角色，而不仅仅是一个工具。

## 二、产品定位

### 一句话定位

**OfferBot 是一个 AI 求职顾问 Agent，帮助求职者从认识自己到拿到 offer 的全流程。**

### 产品定位矩阵

```
                    覆盖环节多
                        ↑
                        │
         OfferBot ──────┤
         (AI Agent)     │
                        │
  简历工具 ─────────────┼──────────── 招聘平台
  (单环节)              │              (平台方)
                        │
         自动投递脚本 ──┤
         (无智能)       │
                        ↓
                    覆盖环节少
  ←─────────────────────┼─────────────────────→
  用户视角                                平台视角
```

OfferBot 的差异化：
- 不是招聘平台（不拥有岗位数据），而是跨平台的求职助手
- 不是单点工具（不只做简历或投递），而是覆盖全流程的 Agent
- 不是无脑脚本（不做无差别投递），而是基于理解的智能决策

### 核心价值主张

> "让 AI 真正理解你，然后帮你找到最合适的工作。"

三个关键词：
1. **理解** — 通过记忆系统持续积累对用户的认知，不是每次从零开始
2. **匹配** — 基于知识图谱和多维度分析，不是简单的关键词匹配
3. **全流程** — 从岗位发现到面试追踪，一个 Agent 覆盖所有环节

## 三、目标用户

### 核心用户画像

| 维度 | 描述 |
|------|------|
| 年龄 | 22-35 岁 |
| 职业阶段 | 初中级技术人员、产品经理、运营等互联网从业者 |
| 求职状态 | 在职看机会 或 离职求职中 |
| 技术素养 | 能使用命令行工具，愿意配置 API Key |
| 核心诉求 | 高效找到匹配的岗位，减少重复劳动 |
| 使用频率 | 求职期间每天使用，非求职期间低频 |

### 用户分层

| 层级 | 用户类型 | 核心需求 | 产品策略 |
|------|----------|----------|----------|
| P0 | 活跃求职者 | 批量筛选 + 匹配分析 + 投递管理 | 核心功能，优先满足 |
| P1 | 被动求职者 | 市场了解 + 机会发现 | 低频使用，知识检索为主 |
| P2 | 开发者/贡献者 | 二次开发 + 学习参考 | 开源社区，文档驱动 |

## 四、产品愿景

### 短期愿景（V1，当前）

一个可用的 AI 求职顾问，覆盖核心求职流程：
- 多平台岗位数据采集（猎聘、智联）
- 基于知识图谱的智能岗位推荐
- 7 维度匹配分析
- 面试漏斗追踪
- 个性化打招呼语生成
- 用户记忆画像系统

### 中期愿景（V2）

一个有"记忆"的求职伙伴：
- 跨求职周期的经验积累（上次求职的教训自动应用到这次）
- 面试复盘与策略优化（基于面试结果反馈调整匹配权重）
- 简历版本管理与 A/B 测试（不同简历版本的回复率对比）

### 长期愿景（V3）

一个求职领域的通用 AI Agent 框架：
- 插件化平台适配（社区贡献新平台的 Skill）
- 多语言支持（海外求职场景）
- 团队协作（猎头/HR 视角的反向匹配）

## 五、产品原则

1. **AI 做决策，人做确认** — Agent 可以自动分析和推荐，但投递等不可逆操作必须经过用户确认
2. **理解优先于效率** — 宁可多花时间建立用户画像，也不做无差别的批量操作
3. **数据属于用户** — 所有数据存储在本地，不上传云端，用户完全控制自己的数据
4. **场景驱动设计** — 先定义用户要做什么，再倒推需要什么功能
5. **渐进式体验** — 新用户可以从简单的岗位搜索开始，随着使用深入逐步解锁高级功能
