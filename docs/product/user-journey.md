# 用户旅程地图

> 版本: 1.0 · 2026.04

## 一、全流程旅程概览

```
阶段:     初识        建立画像       岗位发现       匹配分析       投递行动       面试管理       复盘优化
          │            │              │              │              │              │              │
用户动作: 配置启动     上传简历       搜索岗位       查看匹配度     确认投递       更新状态       查看漏斗
          │            对话交流       浏览推荐       分析差距       设置打招呼语   准备面试       调整策略
          │            │              │              │              │              │              │
Agent:    欢迎引导     提取记忆       爬取数据       7维度评分      生成打招呼语   提供建议       分析转化率
          检查配置     构建画像       构建图谱       排序推荐       执行投递       追踪状态       输出报告
          │            │              │              │              │              │              │
情绪:     好奇         期待           兴奋/疲惫      纠结/清晰      紧张/期待      焦虑/自信      反思/调整
          │            │              │              │              │              │              │
触点:     首页         对话+简历上传  对话+岗位列表  对话+匹配报告  对话+确认弹窗  对话+面试面板  对话+统计面板
```

## 二、阶段详述

### 阶段 1：初识（首次使用）

**用户目标：** 快速了解产品能做什么，完成基础配置。

```
用户打开 OfferBot (localhost:7860)
  │
  ├── 看到欢迎页
  │   ├── 产品简介："我是你的 AI 求职顾问..."
  │   ├── API Key 配置入口（醒目提示）
  │   └── 场景引导卡片（"你可以问什么"）
  │
  ├── 配置 API Key
  │   └── 进入设置页面 → 填入 DashScope API Key → 保存
  │
  └── 点击引导卡片或直接输入
      └── 开始第一次对话
```

**关键体验指标：**
- 从打开页面到开始第一次对话 < 3 分钟
- 引导卡片覆盖最常见的 3 个场景

**设计要点：**
- 未配置 API Key 时，醒目提示但不阻断浏览
- 引导卡片用自然语言而非功能名称（"帮我搜索岗位" 而非 "数据采集"）

---

### 阶段 2：建立画像

**用户目标：** 让 Agent 了解自己的背景和求职意向。

```
用户上传简历（PDF/Markdown）
  │
  ├── 系统自动解析 → 提取结构化字段
  │   ├── 基本信息、教育背景、工作经历
  │   ├── 技术栈、项目经验、核心亮点
  │   └── 写入 resumes 表 + 记忆画像
  │
  └── 用户通过对话补充信息
      ├── "我想找 AI 方向的工作，薪资 30-50K"
      ├── "我不考虑外包公司和 996 的"
      └── "我比较看重技术成长和团队氛围"
          │
          └── MemoryExtractor 异步提取
              ├── job_sprint_goals: "AI 方向，30-50K"
              ├── personal_needs: "不接受外包、996"
              └── values_beliefs: "看重技术成长和团队氛围"
```

**关键体验指标：**
- 简历解析成功率 > 95%（PDF + Markdown）
- 记忆提取准确率 > 90%（直接陈述的信息）

**设计要点：**
- 简历解析是程序性操作，不需要对话交互
- 记忆提取对用户透明，不打断对话流
- 用户可以随时查看和修正 Agent 记住的信息

---

### 阶段 3：岗位发现

**用户目标：** 找到与自己匹配的岗位。

**路径 A：主动搜索**
```
用户: "帮我看看猎聘上有什么 AI 岗位"
  │
  ├── Agent 检查 getjob 服务状态
  │   ├── 服务可用 + 已登录 → 继续
  │   └── 服务不可用 → 提示用户启动服务
  │
  ├── Agent 根据用户画像生成搜索配置
  │   └── 展示配置摘要 → 用户确认/调整
  │
  ├── 启动爬取任务（scrapeOnly 模式）
  │   └── 后台轮询进度 → 完成后通知
  │
  └── 同步数据 + 构建知识图谱
      └── "已爬取 156 个岗位，要我帮你分析吗？"
```

**路径 B：语义推荐**
```
用户: "推荐适合我的岗位"
  │
  ├── Agent 调用 get_user_profile 获取画像
  ├── Agent 调用 rag_query(mode="answer") 语义检索
  └── 返回 Top N 推荐 + 推荐理由
```

**路径 C：条件筛选**
```
用户: "上海 40K+ 的 AI 岗位"
  │
  ├── Agent 调用 query_jobs(city="上海", salary_min=40)
  └── 返回符合条件的岗位列表
```

**关键体验指标：**
- 爬取 100 个岗位 < 10 分钟
- 语义推荐响应 < 15 秒
- SQL 筛选响应 < 3 秒

---

### 阶段 4：匹配分析

**用户目标：** 了解自己和目标岗位的匹配程度，找到差距。

```
用户: "这个岗位适合我吗？" / "分析一下这几个岗位"
  │
  ├── Agent 获取用户画像 + 岗位 JD
  │
  ├── 执行 7 维度匹配分析
  │   ├── 程序化维度（薪资/位置/学历/经验）→ 即时计算
  │   └── LLM 维度（技能/职责/语义）→ AI 评估
  │
  └── 输出匹配报告
      ├── 综合分数: 85/100
      ├── 各维度分数和分析
      ├── 匹配技能: [Python, FastAPI, Agent, LLM]
      ├── 缺失技能: [Kubernetes, 分布式训练]
      └── 建议: "整体匹配度较高，建议补充容器化部署经验"
```

**关键体验指标：**
- 单岗位匹配分析 < 15 秒
- 批量分析 10 个岗位 < 3 分钟

---

### 阶段 5：投递行动

**用户目标：** 向匹配的岗位发送个性化的打招呼语。

```
用户: "帮我投递匹配度 80 分以上的岗位"
  │
  ├── Agent 筛选匹配度达标的岗位
  │
  ├── 为每个岗位生成个性化打招呼语
  │   ├── 基于用户画像（语言风格、核心优势）
  │   ├── 基于岗位 JD（匹配的技能亮点）
  │   └── ≤ 200 字，差异化内容
  │
  ├── 展示投递计划 → 等待用户确认
  │   ├── 岗位列表 + 打招呼语预览
  │   └── 用户可以修改打招呼语或移除某个岗位
  │
  └── 执行投递
      ├── 每次投递间隔 10-30 秒（反检测）
      ├── 实时反馈投递进度
      └── 完成后输出投递报告
```

**关键设计决策：**
- 投递前必须用户确认（不可逆操作）
- 打招呼语可以预览和修改
- 投递失败自动重试，3 次失败后跳过

---

### 阶段 6：面试管理

**用户目标：** 追踪所有投递的面试进度，准备面试。

```
用户: "有没有新的面试邀请？"
  │
  ├── Agent 查询面试漏斗数据
  │   ├── 投递总数: 30
  │   ├── 已读: 18 (60%)
  │   ├── 已回复: 5 (28%)
  │   ├── 约面试: 3 (60%)
  │   └── 进行中: round_1 x2, hr_round x1
  │
  └── 用户更新面试状态
      └── "XX 公司一面过了，约了二面"
          → Agent 更新状态: round_1 → round_2
          → 记录状态变更时间

用户: "帮我准备明天 XX 公司的面试"
  │
  ├── Agent 加载面试准备 Skill
  ├── 获取用户记忆（key_points, personal_thoughts, career_planning）
  ├── 获取目标岗位 JD
  └── 生成针对性面试准备建议
      ├── 可能被问到的问题
      ├── 基于用户真实经验的回答建议
      └── 需要重点准备的知识点
```

---

### 阶段 7：复盘优化

**用户目标：** 分析求职效果，调整策略。

```
用户: "分析一下我的求职情况"
  │
  ├── Agent 输出全局统计
  │   ├── 面试漏斗转化率
  │   ├── 按公司规模/行业/薪资的分布
  │   └── 高回复率岗位的共同特征
  │
  └── Agent 给出策略建议
      ├── "回复率偏低，建议优化打招呼语"
      ├── "面试转化率 0%，建议加强面试准备"
      └── "匹配度 85+ 的岗位回复率明显更高"
```

## 三、关键转化节点

| 节点 | 从 | 到 | 关键动作 | 流失风险 |
|------|----|----|----------|----------|
| 激活 | 首次访问 | 完成配置 | 配置 API Key | 高（配置门槛） |
| 建档 | 完成配置 | 上传简历 | 上传简历/对话建档 | 中（需要准备简历） |
| 首次搜索 | 建档完成 | 发起搜索 | 点击引导卡片 | 低（引导明确） |
| 首次分析 | 有数据 | 查看匹配 | 自然语言请求 | 低 |
| 首次投递 | 分析完成 | 确认投递 | 确认投递计划 | 中（投递是不可逆的） |
| 持续使用 | 首次投递 | 日常使用 | 面试追踪/复盘 | 中（需要持续更新状态） |

## 四、情绪曲线与产品干预

```
情绪
  ↑
  │    ★ 首次推荐命中
  │   ╱ ╲
  │  ╱   ╲  ★ 收到面试邀请
  │ ╱     ╲╱ ╲
  │╱           ╲  ★ 拿到 offer
  │             ╲╱
  │
  │  ↓ 配置繁琐    ↓ 投了没回复    ↓ 面试被拒
  │
  └──────────────────────────────────────→ 时间

产品干预点:
  配置繁琐 → 场景引导卡片，3 分钟上手
  投了没回复 → 漏斗数据 + 策略建议，给用户掌控感
  面试被拒 → 复盘分析 + 鼓励性反馈，保持积极心态
```
