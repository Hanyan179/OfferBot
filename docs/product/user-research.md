# 用户调研与痛点分析

> 版本: 1.0 · 2026.04
> 方法: 自身求职经历 + 社区调研 + 竞品用户评价分析

## 一、调研背景

OfferBot 的产品设计源于真实的求职痛点。作为产品的第一个用户（dogfooding），我在 2026 年初的求职过程中系统性地记录了每个环节的时间消耗、情绪波动和效率瓶颈，并结合脉脉、V2EX、即刻等社区的求职讨论，提炼出以下用户洞察。

## 二、目标用户画像

### 画像 A：在职看机会的技术人员

| 维度 | 描述 |
|------|------|

| 对工具的期望 | 帮我筛选，别让我一个个看；帮我分析匹配度 |

### 画像 B：离职求职中的求职者

### 画像 C：转行求职者

| 维度 | 描述 |
|------|------|
| 典型身份 | 传统开发转 AI 方向 / 后端转全栈 |

## 三、求职全流程痛点地图

```
用户旅程:  认识自己 → 发现岗位 → 分析匹配 → 准备材料 → 投递沟通 → 面试准备 → 追踪管理
              │          │          │          │          │          │          │
痛点等级:    ★★★       ★★★★★     ★★★★      ★★★       ★★★★      ★★★★      ★★★★★
              │          │          │          │          │          │          │
情绪曲线:   迷茫       疲惫       纠结       枯燥       焦虑       紧张       混乱
```

### 痛点 1：信息过载与筛选疲劳（★★★★★）

**现象：**
- 猎聘搜索"AI 工程师 上海"返回 500+ 结果
- 其中 60%+ 与用户实际不匹配（经验不符、薪资不符、方向不符）
- 用户需要逐个点开查看 JD，每个花 3-5 分钟
- 看到第 30 个岗位时已经精疲力竭

**用户原话（社区摘录）：**
> "每天刷两小时 Boss，感觉看了很多但又什么都没看。"
> "猎聘推荐的岗位一半以上跟我不相关，浪费时间。"

**产品机会：** AI 预筛选 + 匹配度排序，让用户只看最相关的 Top N。

### 痛点 2：匹配判断的主观性（★★★★）

**现象：**
- 用户凭直觉判断"这个岗位适不适合我"
- 容易高估或低估自己的匹配度
- 不同岗位之间缺乏可比较的量化标准
- 技能差距分析靠人工对比，容易遗漏

**用户原话：**
> "JD 写的技能我都会一点，但不知道够不够。"
> "投了 50 个，回复了 3 个，不知道是简历问题还是方向问题。"

**产品机会：** 7 维度量化匹配分析，给出可比较的分数和具体的差距说明。

### 痛点 3：多岗位并行管理的混乱（★★★★★）

**现象：**
- 同时投递 20-50 个岗位，每个处于不同阶段
- 用 Excel 手动记录，经常忘记更新
- 不知道整体转化率，无法判断策略是否有效
- 面试时间冲突时手忙脚乱

**用户原话：**
> "投了太多，有些公司约面试我都忘了投过。"
> "不知道自己的回复率是高还是低，没有参照。"

**产品机会：** 面试漏斗自动追踪 + 状态机管理 + 转化率分析。

### 痛点 4：打招呼语的同质化（★★★★）

**现象：**
- 大部分求职者使用平台默认的打招呼语或简单模板
- HR 每天收到大量雷同的消息，回复意愿低
- 个性化打招呼语需要针对每个岗位单独写，耗时

**用户原话：**
> "写了一个模板改改就发，感觉 HR 根本不看。"
> "想针对每个岗位写不同的，但实在没时间。"

**产品机会：** 基于用户画像 + 岗位 JD 自动生成个性化打招呼语。

### 痛点 5：面试准备缺乏针对性（★★★★）

**现象：**
- 面试前不知道该重点准备什么
- 通用面经太泛，与具体岗位关联度低
- 自己的项目经历不知道怎么跟岗位需求对应
- 技术面、HR 面、项目面需要不同的准备策略

**用户原话：**
> "面试前一晚疯狂刷题，但面试官问的都不是刷的那些。"
> "不知道怎么把自己的项目经历跟 JD 要求对应起来。"

**产品机会：** 基于用户记忆（真实经验）+ 岗位 JD 生成针对性面试准备建议。

### 痛点 6：求职过程中的自我认知模糊（★★★）

**现象：**
- 不清楚自己的核心竞争力是什么
- 不知道市场对自己这个方向的需求如何
- 简历上写的和实际能力之间有 gap
- 求职目标不清晰，什么都想试试

**产品机会：** 记忆画像系统，通过持续对话帮用户梳理自我认知。

## 四、需求优先级矩阵

```
        影响面大
            ↑
            │
  岗位筛选 ─┤── 面试追踪
  匹配分析  │
            │
  打招呼语 ─┤── 面试准备
            │
  简历优化 ─┤── 记忆画像
            │
            ↓
        影响面小
  ←─────────┼─────────→
  高频使用              低频使用
```

| 优先级 | 需求 | 理由 |
|--------|------|------|
| P0 | 岗位数据采集与筛选 | 求职的起点，每天都要用 |
| P0 | 匹配度分析 | 解决最大痛点（信息过载），直接影响投递质量 |
| P0 | 面试漏斗追踪 | 解决管理混乱痛点，提供全局视角 |
| P1 | 个性化打招呼语 | 提升回复率，但依赖匹配分析的结果 |
| P1 | 面试准备建议 | 高价值但低频（只在面试前使用） |
| P1 | 简历优化 | 求职初期使用一次，后续低频 |
| P2 | 记忆画像系统 | 长期价值，但短期感知不明显 |
| P2 | 数据导出与分析 | 进阶需求，核心用户才会用 |

## 五、关键用户场景（Jobs to be Done）

### 场景 1：周日下午的批量筛选

> 小王是一个 4 年经验的 AI 工程师，在职看机会。周日下午他有 2 小时空闲时间。
> 他打开 OfferBot，说"帮我看看这周猎聘上新增了哪些 AI 岗位"。
> Agent 自动爬取最新数据，用知识图谱做语义匹配，返回 Top 10 最匹配的岗位，
> 每个附带匹配度分数和差距分析。
> 小王花 20 分钟看完 10 个岗位，选了 5 个让 Agent 生成打招呼语并投递。
> 整个过程从 2 小时缩短到 30 分钟。

### 场景 2：面试前的针对性准备

> 小李收到了一家公司的面试邀请，岗位是"AI 产品经理"。
> 她打开 OfferBot，说"帮我准备明天的面试"。
> Agent 调取她的记忆画像（项目经验、技术要点、职业规划），
> 结合目标岗位的 JD，生成一份针对性的面试准备清单：
> 可能被问到的问题、基于她真实经验的回答建议、需要重点准备的知识点。

### 场景 3：求职复盘与策略调整

> 小张已经投递了 30 个岗位，收到了 5 个面试邀请，但都没有拿到 offer。
> 他打开 OfferBot，说"分析一下我的求职情况"。
> Agent 展示面试漏斗数据：投递→已读率 60%、已读→回复率 25%、回复→面试率 67%、面试→offer 率 0%。
> Agent 分析发现：回复率偏低可能是打招呼语不够个性化，面试转化率为 0 可能需要加强面试准备。
> 基于分析结果，Agent 建议调整策略。

## 六、调研结论

1. 求职是一个**高频、高压、多环节**的场景，用户需要的不是单点工具，而是全流程的智能助手
2. 现有招聘平台的推荐算法是**平台视角**的（优化平台的匹配效率），不是**用户视角**的（优化用户的求职效率）
3. AI Agent 的核心价值不在于"自动化"（自动投递脚本也能做），而在于**"理解"**（基于对用户的深度认知做智能决策）
4. 用户对数据隐私高度敏感（简历、求职意向是敏感信息），**本地部署**是关键的信任基础
5. 开源是获取早期用户和建立信任的最佳策略——用户可以审查代码，确认数据不会被上传
