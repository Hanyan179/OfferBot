# AI 产品决策记录

> 版本: 1.0 · 2026.04
> 本文档记录 OfferBot 开发过程中每个关键的 AI 产品决策，包括背景、备选方案、最终选择和理由。
> 这是面试 AI 产品经理时最核心的展示材料。

---

## 决策 1：Agent 架构选择 — ReAct Loop vs DAG vs 状态机

### 背景
OfferBot 需要一个 Agent 编排架构来协调 30+ 个工具的调用。求职场景的特点是：用户意图多样、执行路径不确定、需要根据中间结果动态调整。

### 备选方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| DAG（有向无环图） | 流程可视化，易于调试 | 无法处理动态分支，新场景需要改图 |
| 状态机 | 状态转换明确，可预测 | 状态爆炸问题，复杂场景难以维护 |
| ReAct Loop（while True） | 动态决策，错误恢复只需 continue | 不可预测，调试需要日志 |

### 最终选择
ReAct Loop，参考 Claude Code 的 `queryLoop()` 设计。

### 理由
1. 求职场景的执行路径高度不确定 — 用户说"帮我找工作"，Agent 需要判断：有没有简历？有没有数据？要不要先爬取？这些判断在运行时才能做出
2. 新功能零成本扩展 — 注册一个新 Tool 就自动可用，不需要修改编排逻辑
3. 错误恢复自然 — 工具调用失败，重试 3 次，失败就跳过继续下一步，不需要额外的错误处理流程

### 产品影响
- 用户体验：Agent 的行为更像"对话"而非"流程"，更自然
- 风险：Agent 可能做出不合理的决策 → 通过 System Prompt 约束 + 关键操作用户确认来兜底

## 决策 2：知识检索方案 — LightRAG vs 纯向量检索 vs SQL

### 背景
OfferBot 需要从数百个岗位中帮用户找到最匹配的。传统方案是向量相似度检索（FAISS），但求职场景有特殊需求：不仅要找"相似的"，还要能回答"哪些公司在招 Agent"、"AI 岗位最需要什么技能"这类聚合问题。

### 备选方案

| 方案 | 语义检索 | 聚合分析 | 实体关系 | 实现成本 |
|------|----------|----------|----------|----------|
| FAISS 向量检索 | ✅ | ❌ | ❌ | 低 |
| SQL 全文检索 | ❌ | ✅ | ❌ | 低 |
| LightRAG（图谱+向量） | ✅ | ✅ | ✅ | 中 |
| 自建知识图谱（Neo4j） | ✅ | ✅ | ✅ | 高 |

### 最终选择
LightRAG，同时保留 SQL 查询作为精确筛选的补充。

### 理由
1. 求职场景需要实体关系 — "这个岗位需要什么技能"、"这家公司还有什么岗位"，这些问题需要实体和关系，纯向量检索做不到
2. 聚合分析是高价值场景 — "市场上最需要什么技能"、"AI 岗位薪资分布"，这些洞察对求职策略调整很有价值
3. LightRAG 开箱即用 — 不需要自己设计图谱 schema，自动提取实体和关系
4. 与 SQL 互补 — 精确条件（城市=上海，薪资>40K）用 SQL，语义理解用 LightRAG

### 产品影响
- 用户可以用自然语言问复杂问题，不局限于关键词搜索
- 图谱构建需要 LLM 调用，有 token 成本 → 只在爬取 JD 后一次性构建

### 路由规则（写入 System Prompt）
```
用户提到具体筛选条件（城市、薪资、公司名）→ query_jobs（SQL）
用户问"适合我的"、"推荐"、"类似的"、"差距"、"趋势" → rag_query（知识图谱）
不确定时 → 优先用 rag_query(mode="answer")
```

## 决策 3：记忆系统设计 — 结构化文件 vs 向量 RAG vs 数据库

### 背景
OfferBot 需要"记住"用户在对话中透露的信息（不仅是简历字段），包括个人想法、语言风格、职业规划等。这些信息需要在后续对话中被调用。

### 备选方案

| 方案 | 检索方式 | 可读性 | 用户可编辑 | 实现成本 |
|------|----------|--------|-----------|----------|
| 向量 RAG | 语义检索 | 低 | 难 | 中 |
| SQLite 表 | SQL 查询 | 低 | 难 | 低 |
| 结构化 Markdown 文件 | 分类加载 | 高 | 容易 | 低 |

### 最终选择
结构化 Markdown 文件，按 10 个维度分类存储。

### 理由
1. 数据特征不适合向量检索 — 用户记忆是原子化短句事实（"目标薪资 30-50K"、"不接受 996"），不是长文档。向量检索在短文本上语义区分度差
2. 数据量可控 — 个人记忆几百条，总共几 KB，现代 LLM 上下文窗口完全装得下，不需要检索
3. 用户可读可编辑 — Markdown 文件用户可以直接打开查看和修改，这对建立信任很重要
4. 分类加载节省 token — 面试准备只需要加载 key_points 和 career_planning，不需要加载 language_style

### 产品影响
- 用户可以直接查看 AI 记住了什么，纠正错误 → 建立信任
- 记忆文件可以跨设备同步（复制文件夹即可）
- 未来数据量增长时，可以在分类内部加向量检索，渐进式升级

## 决策 4：浏览器自动化架构 — Python Playwright vs getjob 服务

### 背景
OfferBot 需要从猎聘、智联等平台爬取岗位数据和执行投递。最初计划用 Python Playwright 直接操作浏览器，后来改为调用外部 Java 服务。

### 备选方案

| 方案 | 开发成本 | 稳定性 | 反检测 | 维护成本 |
|------|----------|--------|--------|----------|
| Python Playwright（自建） | 高 | 低（需要从零处理反检测） | 需要自建 | 高 |
| getjob 服务（Java，已有） | 低（HTTP 调用） | 高（成熟爬虫） | 已有 | 低 |
| 第三方爬虫 API | 低 | 中 | 依赖第三方 | 低 |

### 最终选择
getjob 服务（Java Spring Boot），AI Agent 通过 HTTP API 调用。

### 理由
1. 职责分离 — getjob 负责"手脚"（浏览器操作），AI Agent 负责"大脑"（决策与分析）
2. 复用成熟代码 — getjob 已经有完整的反检测、多平台支持、错误处理
3. AI Agent 的价值不在浏览器操作 — 而在于用记忆系统理解用户、用 LLM 做智能匹配、生成个性化策略
4. 独立部署和扩展 — getjob 可以独立升级，不影响 AI Agent

### 产品影响
- 用户需要同时运行两个服务（OfferBot + getjob）→ 通过启动脚本简化
- 架构更清晰，但部署复杂度增加 → 提供 docker-compose 方案

## 决策 5：匹配分析方案 — 纯 LLM vs 纯程序化 vs 混合

### 背景
OfferBot 需要评估用户简历与岗位 JD 的匹配度。这是产品的核心价值之一。

### 备选方案

| 方案 | 准确度 | 成本 | 速度 | 可解释性 |
|------|--------|------|------|----------|
| 纯 LLM 评估 | 高 | 高（每次调用 LLM） | 慢 | 高 |
| 纯程序化规则 | 中 | 零 | 快 | 高 |
| 混合方案 | 高 | 中 | 中 | 高 |

### 最终选择
混合方案：7 个维度中，4 个用程序化计算（薪资/位置/学历/经验），3 个用 LLM 评估（技能/职责/语义）。

### 理由
1. 程序化维度零 token 成本 — 薪资区间重叠、城市匹配、学历对比，这些用规则就能精确计算，不需要 LLM
2. LLM 维度处理模糊匹配 — "3 年 Python 经验"和"熟悉 Python 开发"的匹配度，需要语义理解
3. 批量分析时成本可控 — 分析 50 个岗位，程序化维度瞬间完成，只有 3 个维度需要 LLM 调用
4. 可解释性 — 每个维度独立评分，用户能看到"技能匹配 85 分，薪资匹配 100 分，经验匹配 60 分"

### 产品影响
- 用户得到的不是一个黑盒分数，而是可理解的多维度分析
- 批量分析 10 个岗位 < 3 分钟，用户体验可接受

## 决策 6：投递策略 — 全自动 vs 半自动 vs 纯手动

### 背景
OfferBot 可以自动化投递流程，但投递是不可逆操作（发出去的打招呼语无法撤回）。

### 备选方案

| 方案 | 效率 | 风险 | 用户控制感 |
|------|------|------|-----------|
| 全自动（无人值守） | 最高 | 高（可能投递不合适的岗位） | 低 |
| 半自动（AI 推荐 + 用户确认） | 中 | 低 | 高 |
| 纯手动（只提供分析） | 低 | 最低 | 最高 |

### 最终选择
半自动：AI 筛选 + 生成打招呼语 + 展示投递计划 + 用户确认后执行。

### 理由
1. 产品原则"AI 做决策，人做确认" — 投递是不可逆操作，必须用户确认
2. 用户信任需要时间建立 — 初期用户不会完全信任 AI 的判断，需要看到投递计划后才放心
3. 打招呼语需要预览 — 用户可能想修改某些表述
4. 避免平台封禁 — 全自动无差别投递容易触发平台风控

### 产品影响
- 投递效率比全自动低，但用户满意度和信任度更高
- 未来可以根据用户信任度逐步放开自动化程度

## 决策 7：Skills 系统定位 — 硬编码流程 vs 场景参考规范

### 背景
OfferBot 有多个业务场景（简历生成、面试准备、打招呼语等），需要一种方式组织场景逻辑。

### 备选方案

| 方案 | 灵活性 | AI 自主性 | 维护成本 |
|------|--------|----------|----------|
| 硬编码流程（if-else） | 低 | 低（按固定步骤执行） | 高 |
| Workflow 编排（DAG） | 中 | 低（按图执行） | 中 |
| 场景参考规范（Skill） | 高 | 高（AI 自主决策） | 低 |

### 最终选择
场景参考规范：Skill 提供上下文和示例，AI 基于这些信息自主决策。

### 理由
1. AI 的判断力足够 — 现代 LLM 给足上下文后，能做出合理的执行决策
2. 新场景零代码 — 写一个 SKILL.md 文件就能添加新场景，不需要改代码
3. 社区可贡献 — 开源后，社区可以贡献新的 Skill 文件
4. 容错性好 — AI 可以根据实际情况灵活调整，不会因为某个步骤失败就整个流程卡住

### 产品影响
- Skill 是 OfferBot 的"知识包"，可以不断积累和扩展
- 开源社区可以贡献行业特定的 Skill（如"前端面试准备"、"产品经理面试准备"）

## 决策 8：数据存储 — 云端 vs 本地

### 背景
OfferBot 处理的数据包括简历、求职意向、面试记录、记忆画像，都是高度敏感的个人信息。

### 备选方案

| 方案 | 便利性 | 隐私 | 成本 | 离线可用 |
|------|--------|------|------|----------|
| 云端存储（SaaS） | 高 | 低 | 需要服务器 | 否 |
| 本地存储 | 中 | 高 | 零 | 是 |
| 混合（本地+可选云同步） | 高 | 中 | 可选 | 部分 |

### 最终选择
纯本地存储：SQLite + Markdown 文件 + JSONL。

### 理由
1. 求职数据极度敏感 — 简历、薪资期望、面试记录，泄露后果严重
2. 开源项目的信任基础 — 用户可以审查代码，确认没有数据上传行为
3. 零运维成本 — 不需要服务器，不需要数据库运维
4. 离线可用 — 除了 LLM API 调用，其他功能都可以离线使用

### 产品影响
- 用户数据完全自主，可以随时删除
- 跨设备同步需要用户手动复制数据文件夹 → 未来可以加可选的加密云同步
- 无法做跨用户的数据分析（如行业薪资趋势）→ 可以通过匿名聚合数据解决

## 决策 9：前端方案 — 自建 UI vs Chainlit vs Gradio

### 背景
OfferBot 需要一个 Web 界面，核心是对话交互，辅以数据展示面板。

### 备选方案

| 方案 | 开发成本 | 定制性 | 对话体验 | 数据展示 |
|------|----------|--------|----------|----------|
| 自建 React UI | 高 | 最高 | 需要自建 | 完全自定义 |
| Gradio | 低 | 低 | 一般 | 组件有限 |
| Chainlit | 低 | 中 | 优秀 | 需要补充 |

### 最终选择
Chainlit（对话 UI）+ FastAPI（管理页面）。

### 理由
1. 对话是核心交互 — Chainlit 的对话体验开箱即用，支持流式输出、工具调用展示、文件上传
2. 快速验证产品 — 不需要花时间在前端开发上，聚焦 Agent 核心能力
3. FastAPI 补充管理页面 — 设置、岗位列表、统计等非对话场景用 Jinja2 模板
4. 未来可替换 — 如果需要更好的 UI，可以换成自建前端，Agent 后端不受影响

### 产品影响
- 对话体验好，但数据可视化能力有限 → V1.5 计划增强
- 部署简单，一个命令启动

## 总结：AI 产品决策的核心原则

通过以上 9 个决策，可以提炼出 OfferBot 的 AI 产品决策原则：

1. **场景决定技术** — 不是"用什么技术"，而是"场景需要什么能力"。记忆系统不用 RAG 是因为数据特征不匹配，不是因为不会做 RAG。
2. **渐进式复杂度** — 先用最简单的方案跑通，数据量增长后再升级。记忆系统先用文件，未来可以加向量检索。
3. **人机协作而非替代** — AI 做分析和推荐，人做确认和决策。投递必须用户确认，不做无人值守。
4. **可解释性优先** — 用户需要理解 AI 为什么这么推荐。7 维度匹配分数比一个黑盒分数更有价值。
5. **数据主权不可妥协** — 本地存储是底线，即使牺牲便利性。
