## 场景流转图

```
用户进入对话
  │
  ├── 新用户（无画像）
  │     → 引导建立画像（子Agent）→ 引导数据采集（S2）→ 进入正常对话
  │
  ├── 老用户（有画像，无数据）
  │     → 引导数据采集（S2）→ 进入正常对话
  │
  └── 老用户（有画像，有数据）
        → 直接进入正常对话
              │
              ├── "推荐适合我的岗位" ──────→ S3 知识检索（核心工具）
              ├── "上海40K+的AI岗位" ──────→ S3 条件筛选（核心工具）
              ├── "这个岗位适合我吗" ──────→ S4 匹配分析（核心工具）
              ├── "帮我爬取岗位" ──────────→ S2 数据采集（注入采集集）
              ├── "帮我投递这个岗位" ──────→ S5 求职行动（注入投递集）
              ├── "我投了多少" ────────────→ S6 进度追踪（核心工具）
              └── "聊聊职业规划" ──────────→ S7 日常对话（无需工具）
```

## 路由机制

工具/Skill/Prompt 片段总共二三十个，不需要向量化匹配。路由方式由 LLM 自身的 function calling 能力决定 — 核心集始终可用，场景集通过 `activate_toolset` 元工具按需加载。

具体机制在架构设计阶段确定，场景定义只关注"什么场景需要什么能力"。

## 设计原则

- 场景驱动：先定义用户要做什么，再倒推需要什么工具/Skill/Prompt
- 精准注入：每个场景只注入它需要的资源，不多不少
- 主 Agent 轻量化：主 Agent 只做对话调度，具体能力由场景模块提供

## 场景总览

| 编号 | 场景 | 类型 | 主 Agent 是否直接处理 |
|------|------|------|----------------------|
| S1 | 用户画像 | 子 Agent | 否，子 Agent 负责 |
| S2 | 数据采集 | 工具场景 | 是，需要工具集 |
| S3 | 知识检索与岗位发现 | 核心场景 | 是，核心工具 |
| S4 | 匹配分析 | 核心场景 | 是，核心工具 |
| S5 | 求职行动 | 工具场景 | 是，需要工具集 |
| S6 | 进度追踪 | 数据查询 | 是，轻量查询 |
| S7 | 日常对话 | 纯对话 | 是，无需工具 |

## S1：用户画像（子 Agent 实现）

**核心思路**：主 Agent 不需要知道画像怎么构建的，只需要能获取画像结果。

### 实现方式
- 子 Agent 负责：对话中提取用户信息 → 分类存储 → 构建画像
- 主 Agent 只需要：
  - `get_user_cognitive_model()` — 获取分类画像信息
  - 按需获取单维度详情（如只要技能维度、只要求职目标维度）

### 注入策略
- 默认注入：用户摘要（轻量，几百 token）-默认注入用户的结构化画像。
- 按需注入：当场景需要详细信息时，获取单维度信息
- 不注入：画像构建过程的工具（save_memory、update_memory 等由子 Agent 管理）

### 简历解析
- 属于程序性功能，不是对话场景
- 用户上传文件 → 系统自动解析 → 写入画像
- 不需要 LLM 参与决策

---

## S2：数据采集

**前提**：本地没有数据，或用户需要新数据时触发。

### 子场景

#### S2.1 搜索新岗位（启动爬取任务）
- 需要知道：如何启动 Getjob 服务、如何根据用户画像构建搜索条件
- 工具：`platform_start_task`, `platform_stop_task`, `platform_status`
- 前置条件：Getjob 服务可用

#### S2.2 获取岗位详情（爬取 JD）
- 场景：本地有岗位列表但缺 JD
- 工具：`fetch_job_detail`
- 后置动作：爬取完成后自动向量化/图谱化（LightRAG insert）

#### S2.3 数据同步
- 场景：Getjob 爬取完成后同步到本地数据库
- 工具：`sync_jobs`

### 注入策略
- 这些工具只在用户明确要求采集数据时注入
- 日常对话、岗位推荐等场景不需要这些工具

---

## S3：知识检索与岗位发现

**核心场景，基于已爬取数据的知识匹配。**

### 子场景

#### S3.1 条件筛选
- "上海 40K+ AI 岗位"、"3年经验的后端"
- 工具：`query_jobs`（SQL 查询）

#### S3.2 语义推荐
- "适合我的岗位"、"推荐一下"
- 工具：`rag_query(mode="answer")`，需要先获取用户画像

#### S3.3 相似推荐
- "找跟这个类似的"
- 工具：`rag_query(mode="search")`

#### S3.4 知识问答
- "AI 岗位技能需求 TOP10"、"哪些公司在招 Agent"
- 工具：`rag_query(mode="answer")`
- 基于图谱聚合能力

#### S3.5 精确查找
- "京东有什么岗位"
- 工具：`query_jobs`（company 参数）

### 注入策略
- `query_jobs` 和 `rag_query` 属于核心工具，始终注入
- 前提：本地有数据。如果没有数据，应引导用户先采集（→ S2）

---

## S4：匹配分析

**基于已爬取数据的综合分析。**

### 子场景

#### S4.1 匹配度分析
- "这个岗位适合我吗"、"分析一下差距"
- 需要：用户画像 + 岗位 JD（来自 rag_query 或 query_jobs）
- 工具：`get_user_profile` + `rag_query(mode="answer")`

#### S4.2 技能差距分析
- "我缺什么技能"
- 需要：用户画像 + 岗位 JD 图谱中的技能实体
- 工具：`get_user_profile` + `rag_query(mode="answer")`

#### S4.3 技能 TOP 分析
- "市场上最需要什么技能"
- 基于图谱聚合
- 工具：`rag_query(mode="answer")`

#### S4.4 批量对比
- "这几个岗位哪个最适合我"
- 需要：用户画像 + 多个岗位信息
- 工具：`get_user_profile` + `rag_query(mode="answer")`

### 注入策略
- 核心工具（rag_query、get_user_profile）始终可用
- 这个场景不需要额外工具注入

---

## S5：求职行动

**基于已有数据，通过浏览器自动化执行实际操作。**

### 子场景

#### S5.1 岗位投递
- 基于浏览器对应的单一页面 ID，为用户执行投递
- 不需要简历定制（投递是第三方平台，不能随时修改简历）
- 工具：`platform_deliver`
- 前置：需要知道投递哪个岗位（岗位 ID）

#### S5.2 个性化打招呼
- 基于用户画像 + 岗位信息生成打招呼语
- 工具：`get_user_profile` + `rag_query` + `platform_deliver`
- 这是一个 Skill 场景（简历生成类似的执行逻辑）

### 不包含
- ~~简历定制~~（第三方平台不支持动态修改）
- ~~面试准备~~（不属于 Agent 自动化范畴）

### 注入策略
- 投递相关工具只在用户明确要求投递时注入
- 打招呼语生成可以作为 Skill 实现

---

## S6：进度追踪

**基于本地数据的状态查询。**

### 子场景

#### S6.1 投递状态
- "我投了哪些"、"投递进度"
- 工具：`get_stats`, `get_interview_funnel`

#### S6.2 面试追踪
- "有没有约面试的"
- 工具：`get_interview_funnel`

#### S6.3 统计概览
- "整体情况怎么样"
- 工具：`get_stats`, `job_count`

### 注入策略
- `get_stats` 和 `job_count` 属于核心工具，始终可用
- `get_interview_funnel` 可以始终可用（轻量查询）

---

## S7：日常对话

- 职业规划建议、市场趋势讨论、闲聊鼓励
- 不需要工具，纯 LLM 对话
- 如果涉及具体数据（"市场趋势"），可能触发 S3 的知识问答

## 场景 → 资源映射

### 核心工具（始终注入，~6 个）

| 工具 | 用途 | 覆盖场景 |
|------|------|---------|
| `get_user_profile` | 获取用户画像 | S3, S4, S5 |
| `query_jobs` | SQL 条件查询 | S3.1, S3.5 |
| `rag_query` | 知识图谱检索 | S3.2, S3.3, S3.4, S4 |
| `get_stats` | 统计概览 | S6 |
| `job_count` | 岗位数量 | S6 |
| `get_interview_funnel` | 面试漏斗 | S6 |

### 场景工具（按需注入）

| 工具集 | 工具 | 触发场景 |
|--------|------|---------|
| 采集集 | `fetch_job_detail`, `platform_start_task`, `platform_stop_task`, `sync_jobs`, `platform_status` | S2 |
| 投递集 | `platform_deliver`, `save_application` | S5 |
| 管理集 | `platform_config`, `platform_update_config`, `getjob_service_manager`, `platform_stats` | 运维操作 |
| Web 集 | `web_fetch`, `web_search` | 需要外部信息时 |

### 子 Agent 管理（主 Agent 不直接使用）

| 工具 | 管理者 | 说明 |
|------|--------|------|
| `save_memory` | 记忆子 Agent | 对话后自动提取 |
| `update_memory` | 记忆子 Agent | |
| `delete_memory` | 记忆子 Agent | |
| `get_memory` | 记忆子 Agent / 主 Agent 按需 | |
| `search_memory` | 记忆子 Agent / 主 Agent 按需 | |
| `get_user_cognitive_model` | 主 Agent 可用 | 获取摘要 |
| `update_user_profile` | 简历解析程序 / 子 Agent | |

### 不再需要的工具

| 工具 | 原因 |
|------|------|
| `save_job` | 程序性操作，不需要 LLM 决策 |
| `export_csv` | UI 功能，不是对话场景 |
| `add_to_blacklist` / `remove_from_blacklist` | 低频，可以通过 UI 操作 |
| `delete_jobs` | 管理操作，不是对话场景 |
| `get_skill_content` | Skill 内容应该由路由层自动注入，不需要 LLM 主动获取 |
