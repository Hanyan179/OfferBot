# MooBot 设计思考与架构解析

> 一个 AI 时代的产品开发方法论实践。从想法到落地，记录整个思考过程。

---

## 一、开发方法论：AI 时代怎么做产品

传统软件开发是"需求 → 设计 → 编码 → 测试"的线性流程。AI 时代不一样——你不知道 LLM 能做到什么程度，不知道 prompt 写成什么样效果最好，不知道工具调用的准确率有多高。所以我的方法是：**先看到，再想清楚，再做对，再做好。**

### 第一步：基于想法，先做 UI 可视化

不写一行后端代码，先把界面搭出来。

为什么？因为 AI 产品最大的风险不是"做不出来"，而是"做出来没人用"。先有界面，你才能：
- 感受到用户的操作路径是什么
- 发现哪些功能是伪需求
- 跟自己（或别人）对齐"这个东西到底长什么样"

MooBot 的做法：用 Chainlit 搭对话 UI + FastAPI 渲染管理页面，先跑起来一个能看到的东西。对话框、岗位列表、设置页面、知识图谱可视化——先有壳，再填肉。

### 第二步：架构设计思考

UI 跑起来之后，你会自然地开始想：
- 用户说一句话，系统要做哪些事？
- 数据从哪来，存到哪去？
- 哪些是 LLM 该做的，哪些是程序该做的？

这一步的核心产出是**架构分层**：

```
用户层 (Browser)
  ↓
Web 层 (FastAPI + Chainlit)
  ↓
Agent 核心引擎层 (Executor + LLM + Tools)
  ↓
Tool 层 (30+ 工具)
  ↓
数据持久层 (SQLite + 记忆文件 + LightRAG)
  ↓
外部服务 (LLM API + Getjob 爬虫服务)
```

关键决策在这一步做出：自建 Agent Loop 还是用框架？用什么数据库？RAG 怎么做？这些决策一旦定下来，后面的开发就是填空。

### 第三步：初步流程跑通

不追求完美，先让核心链路跑通：
- 用户说话 → LLM 回复 ✅
- LLM 调用工具 → 拿到结果 → 继续对话 ✅
- 数据存进数据库 → 能查出来 ✅

这一步验证的是：**架构设计是否可行**。如果核心链路跑不通，说明架构有问题，趁早改。

### 第四步：场景化编排

核心链路通了之后，开始思考：**Agent 需要满足用户的哪些场景？每个场景的执行流程是什么？**

这是最关键的一步。不是"我有什么工具"，而是"用户要做什么事"。从用户场景倒推需要什么能力。

MooBot 定义了 7 个核心场景（详见下文），每个场景有明确的触发条件、需要的工具、执行流程。

### 第五步：场景化路由设计与测试

场景定义好之后，下一个问题是：**怎么让 Agent 知道当前是什么场景，该用什么工具？**

这是一种路由层思考：
- 31 个工具全塞给 LLM？→ 工具太多，LLM 会选错
- 按场景分组，按需注入？→ 谁来决定注入哪组？
- LLM 自己判断场景？→ 准确率够吗？

路由设计完成后，需要逐个场景测试路由是否正确。这不是单元测试，是**路由流程测试**——给一句用户输入，验证 Agent 是否选对了场景和工具。

### 第六步：技术性实现

场景和路由确定后，才进入具体的技术实现：
- 这些场景需要哪些技术？（RAG、知识图谱、浏览器自动化、记忆系统）
- 哪些技术是通用的，哪些是场景特有的？
- 每个技术点的实现方案是什么？

然后逐个攻克技术点，每完成一个就跑一次真实流程测试。

### 第七步：整体流程测试

所有技术点做完后，端到端跑一遍完整流程：
- 新用户进来 → 建立画像 → 搜索岗位 → 匹配分析 → 投递 → 追踪
- 老用户回来 → 加载记忆 → 继续上次的进度

这一步会暴露大量集成问题。

### 第八步：运维优化调优

真实跑起来之后，才是真正的优化阶段：
- **幻觉率**：Agent 是否编造了不存在的岗位？是否虚构了匹配分析？
- **召回率**：RAG 检索是否漏掉了相关岗位？
- **循环问题**：Agent 是否在工具调用中陷入死循环？
- **Token 消耗**：System Prompt 是否太长？工具结果是否需要截断？
- **延迟**：每轮对话的响应时间是否可接受？
- **路由准确率**：Agent 是否选对了场景和工具？

这些都是在真实使用中才能发现和优化的问题，不是设计阶段能预见的。

## 二、Agent 架构（基于真实代码）

### 2.1 核心引擎：ReAct Loop

MooBot 的心脏是 `Executor.chat()`（`agent/executor.py`），一个参考 Claude Code `queryLoop()` 设计的 ReAct 循环：

```
用户消息 → Executor.chat()
  │
  │  ┌─── while turn < max_turns ───┐
  │  │                               │
  │  │  ① LLM 调用 (chat_with_tools) │
  │  │     → 文本回复 + tool_calls   │
  │  │                               │
  │  │  ② 有 reasoning → yield 思考  │
  │  │  ③ 有 text → yield 回复       │
  │  │  ④ 无 tool_calls → 结束       │
  │  │                               │
  │  │  ⑤ 执行每个 tool_call         │
  │  │     → 指数退避重试 (2s/4s/8s) │
  │  │     → 结果追加到消息历史       │
  │  │                               │
  │  │  turn++ → 回到 ①              │
  │  └───────────────────────────────┘
  │
  └→ yield AgentEvent 给 UI 层
```

关键设计决策：
- **不用 LangChain/LangGraph** — 自己写的 while 循环更轻量、更可控，出了问题能直接 debug
- **不用 Planner** — LLM 直接在对话中决策，不需要预先生成执行计划
- **OpenAI function calling** — 模型可以在一次回复中同时输出文本和工具调用
- **工具结果截断** — 超过 2000 字符的结果自动提取摘要，控制 token 消耗

### 2.2 启动引导链

`bootstrap.py` 负责初始化所有组件：

```
bootstrap(db, api_key, model, base_url)
  ├── create_tool_registry()  → 注册 31 个 Tool
  ├── LLMClient()             → OpenAI 兼容格式，支持多厂商
  ├── GetjobClient()          → 爬虫服务 HTTP 客户端
  ├── JobRAG()                → LightRAG 知识图谱
  ├── Planner()               → 任务规划器（备用）
  ├── Executor()              → ReAct 执行器
  └── SkillLoader()           → 场景剧本加载器
```

### 2.3 System Prompt 构成

`system_prompt.py` 拼接完整的 System Prompt：

```
基础人设（求职顾问身份 + 行为准则）
  + RAG 检索策略（路由规则：什么场景用什么工具）
  + 记忆系统指引（记忆工具使用方法）
  + Skills 摘要（场景剧本列表，按需加载完整内容）
```

### 2.4 记忆系统：子 Agent 异步提取

`memory_extractor.py` 是一个独立的子 Agent，在对话结束后异步运行：

```
对话结束 → MemoryExtractor.extract()
  ├── 预读已有记忆摘要
  ├── 构建提取 Prompt（对话内容 + 分类列表 + 已有记忆）
  └── 多轮工具调用循环（最多 5 轮）
       ├── get_memory → 查看已有记忆
       ├── save_memory → 新增
       ├── update_memory → 更新（避免重复）
       └── delete_memory → 清理冗余
```

核心原则：**更新优先，避免膨胀**。记忆系统维护的是精炼的用户画像，不是对话日志。

### 2.5 Skills 系统：场景剧本

`skills/` 目录下每个文件夹是一个 Skill，包含 `SKILL.md`（YAML frontmatter + Markdown 正文）。

System Prompt 只注入摘要（名称 + 描述 + 适用场景），完整内容通过 `get_skill_content` 按需加载。这样做的好处是节省 token——大部分对话不需要所有 Skill 的完整内容。

当前 Skills：
| Skill | 场景 |
|-------|------|
| 猎聘岗位采集与投递 | 爬取、投递的完整操作流程 |
| 简历生成 | 简历生成与优化 |
| 定制简历 | 针对特定岗位定制简历 |
| 面试准备 | 面试准备建议 |
| 模拟打招呼 | 生成个性化打招呼语 |

### 2.6 Tool 层：31 个工具，4 个分类

| 分类 | 工具数 | 代表工具 |
|------|--------|---------|
| Data（本地数据操作） | 17 | query_jobs, save_job, memory_tools, user_profile, stats |
| Getjob（爬虫服务交互） | 10 | platform_start/stop, fetch_detail, sync_jobs, deliver |
| Browser（网页操作） | 2 | web_fetch, web_search |
| AI（智能辅助） | 1 | get_skill_content |

### 2.7 数据层

| 存储 | 用途 |
|------|------|
| SQLite (WAL) | 岗位、简历、投递、面试、配置 |
| Markdown 文件 | 用户记忆画像（10 个分类） |
| LightRAG | 知识图谱 + 向量检索（岗位 JD 图谱化） |
| JSONL | 对话历史 |

## 三、场景化设计

### 3.1 场景总览

从用户视角出发，MooBot 需要覆盖 7 个核心场景：

| 编号 | 场景 | 触发示例 | 核心工具 |
|------|------|---------|---------|
| S1 | 用户画像 | 上传简历、聊天中透露信息 | 子 Agent 自动处理 |
| S2 | 数据采集 | "帮我搜岗位"、"爬取详情" | platform_start, fetch_detail, sync_jobs |
| S3 | 知识检索 | "推荐适合我的"、"上海 AI 岗位" | query_jobs, rag_query |
| S4 | 匹配分析 | "这个岗位适合我吗" | get_user_profile + rag_query |
| S5 | 求职行动 | "帮我投递" | platform_deliver |
| S6 | 进度追踪 | "投了多少"、"面试进度" | get_stats, get_interview_funnel |
| S7 | 日常对话 | "聊聊职业规划" | 无需工具 |

### 3.2 场景流转

```
用户进入对话
  │
  ├── 新用户（无画像）→ S1 引导建立画像 → S2 引导采集数据 → 正常对话
  ├── 老用户（有画像，无数据）→ S2 引导采集 → 正常对话
  └── 老用户（有画像，有数据）→ 直接进入正常对话
        │
        ├── "推荐适合我的岗位" ──→ S3 知识检索
        ├── "上海 40K+ AI 岗位" ──→ S3 条件筛选
        ├── "这个岗位适合我吗" ──→ S4 匹配分析
        ├── "帮我爬取岗位" ────→ S2 数据采集
        ├── "帮我投递" ────────→ S5 求职行动
        ├── "投了多少" ────────→ S6 进度追踪
        └── "聊聊职业规划" ────→ S7 日常对话
```

### 3.3 三层检索架构

岗位检索是最核心的场景，采用三层架构：

| 检索方式 | 工具 | 适用场景 | Agent 消费 |
|---------|------|---------|-----------|
| SQL 条件查询 | query_jobs | "上海 40K+"、"京东的岗位" | ❌ 推前端 |
| 知识图谱语义检索 | rag_query(mode="answer") | "适合我的"、"技能 TOP10" | ✅ Agent 回答 |
| 图谱实体检索 | rag_query(mode="search") | "类似的岗位" | ❌ 推前端 |

路由判断规则：
1. 有具体筛选条件（城市、薪资、公司名）→ query_jobs
2. 问"适合我的"、"推荐"、"差距"、"趋势" → rag_query
3. 不确定时 → 优先 rag_query(mode="answer")

## 四、路由层：当前卡点与目标

### 4.1 现状问题

当前 31 个工具全部平铺注册，每次 LLM 调用都看到全部工具。这导致：
- LLM 选错工具的概率增大
- System Prompt 臃肿（所有场景策略堆在一起）
- 子 Agent 的工具（记忆写入）暴露给主 Agent

### 4.2 目标架构

```
核心工具集（始终可用，~6 个）
  get_user_profile, query_jobs, rag_query,
  get_stats, job_count, get_interview_funnel

场景工具集（按需注入）
  采集集: fetch_detail, platform_start/stop, sync_jobs
  投递集: platform_deliver, save_application
  管理集: platform_config, service_manager
  Web 集: web_fetch, web_search

子 Agent 专用（主 Agent 不可见）
  save_memory, update_memory, delete_memory
```

## 五、技术选型与理由

| 技术 | 为什么用 |
|------|---------|
| 自建 Agent Loop | 求职场景任务链不固定，需要 LLM 动态决策；自己写的循环更可控 |
| OpenAI SDK | 统一接口兼容多家 LLM（DashScope/OpenAI/Gemini/DeepSeek） |
| LightRAG | 知识图谱 + 向量双模检索，JD 图谱化后支持技能关系匹配 |
| SQLite (WAL) | 单用户本地工具，不需要服务端数据库 |
| Playwright (getjob) | 招聘平台没有开放 API，只能浏览器自动化 |
| Chainlit | 对话 UI（流式输出、Markdown、工具调用展示） |
| FastAPI | API 路由 + 管理页面渲染 |
| Markdown 记忆文件 | 人类可读、可编辑、Git 友好 |

## 六、经验总结

### 做对的事
- **先 UI 后逻辑**：避免了大量"做出来发现不对"的返工
- **自建 Agent Loop**：比用框架灵活得多，出问题能直接 debug
- **记忆系统用子 Agent**：主对话不受影响，记忆提取质量更高
- **Skills 按需加载**：节省 token，System Prompt 不会无限膨胀

### 踩过的坑
- **工具太多导致幻觉**：31 个工具全塞给 LLM，选错率明显上升
- **System Prompt 臃肿**：所有场景策略堆在一起，LLM 反而不知道该关注什么
- **缺少路由层**：方法都有了，但"谁在什么时候被调用"的调度逻辑缺失
- **真实测试不够**：单元测试通过不代表真实场景能用

### 下一步
- 实现路由层（核心集 + 场景集按需注入）
- 接入 Langfuse 评测平台，量化幻觉率和路由准确率
- 逐个场景端到端测试
